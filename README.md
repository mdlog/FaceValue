# FaceValue — Regulated, Private Ticket Resale (ZK on Stellar)

> Submission for **Stellar Hacks: Real-World ZK** (Stellar Development Foundation).
> Prove every ticket resale is **at or below the public face-value cap** — without
> revealing buyer, seller, or price — and let a regulator reconstruct the detail
> via a scoped **view key**. The cap is enforced by a zero-knowledge proof verified
> in a Soroban contract, **not** by trusting a platform.

**Positioning:** enforcement is the headline, privacy is the wedge. This is a
*compliance-enforcement primitive*, not "another zk ticket."

---

## Why ZK is load-bearing (not decorative)

One Circom + Groth16 circuit proves, for each resale:

```
assert  resale_price ≤ per_event_cap        // the regulated cap (public)
assert  ticket ∈ merkle(issued_set)         // a genuine issued ticket
emit    nullifier = Poseidon(ticket_secret) // burns the old ticket (anti double-sell)
```

`resale_price`, the buyer/seller, and the ticket secret stay **private**. The
range check is a hard constraint (`LessEqThan(32).out === 1`): an over-cap price
makes the witness **unsatisfiable**, so the prover **cannot even produce** a
proof — there is no proof to verify, on-chain or off. A fair price clears the
cap, the proof verifies, and the nullifier burns the old ticket. Remove the
proof and you cannot enforce the cap privately; that is its load-bearing role.

The verifier reuses Stellar's `groth16_verifier` (Protocol 25/26 BN254 + Poseidon
host functions), so on-chain verification is cheap.

## What's REAL vs MOCK (disclosed honestly)

The **Circom circuit and the Groth16 proving/verification path are now real.**
The resale page generates an actual zero-knowledge proof: a price above the cap
is **unsatisfiable**, so the proof literally fails to generate — that failure is
the enforcement, not a scripted `if`. The product front-end renders the full
user journey on top of that real path.

### Build the ZK artifacts

```bash
npm install
npm run zk:build      # compiles circuits/resale.circom, runs the Groth16 setup,
                      # and emits the artifacts the app reads:
                      #   public/circuits/resale.wasm
                      #   public/circuits/resale_final.zkey
                      #   public/circuits/verification_key.json
                      # plus the demo fixtures: src/lib/zk/fixtures.json
```

The `/api/prove` and `/api/verify` route handlers (Node runtime) load these
artifacts from `public/circuits/`. Without them, the resale page falls back to
the simulated verdict.

### Demo path (the 2–3 min video)
1. **/resale** — press **Scalper · $210** → *Generate proof & submit* → the contract **DENIES** it (above cap), with a tx hash.

## Connect a wallet & submit on-chain (real Soroban verify_resale)

The resale verdict can now be verified on Soroban testnet instead of simulated. A **Connect** button (using Stellar Wallets Kit) appears when a contract is deployed; it signs and submits the real `verify_resale` transaction. The app gracefully falls back to the existing simulated verdict if no contract is configured — full feature-parity, zero breaking changes.

### Setup

1. **Environment**: Dependencies are already in `package.json` (`@stellar/stellar-sdk`, `@creit.tech/stellar-wallets-kit`). Just `npm install`.

2. **Copy env template**:
   ```bash
   cp .env.example .env.local
   ```
   The three Soroban vars are pre-populated with testnet defaults:
   - `NEXT_PUBLIC_FACEVALUE_CONTRACT_ID` — empty until you deploy; this gates the on-chain flow
   - `NEXT_PUBLIC_SOROBAN_RPC_URL` — defaults to `https://soroban-testnet.stellar.org`
   - `NEXT_PUBLIC_STELLAR_NETWORK_PASSPHRASE` — defaults to `Test SDF Network ; September 2015`

3. **Deploy the contract** (requires `stellar` CLI + a funded testnet identity):
   ```bash
   node scripts/soroban/deploy.mjs
   ```
   This script:
   - Builds the Soroban contract
   - Deploys it to testnet
   - Initializes it with the serialized verifying key from your circuit
   - Registers the three events (cap-change, nullifier-burned, verdict-emitted) using cap + root from `src/lib/zk/fixtures.json`
   - Prints the new `NEXT_PUBLIC_FACEVALUE_CONTRACT_ID` to paste into `.env.local`

   (If you don't have `stellar` CLI or a funded testnet identity yet, skip this step and run the app in graceful-fallback mode; see below.)

4. **Paste CONTRACT_ID** into `.env.local`:
   ```
   NEXT_PUBLIC_FACEVALUE_CONTRACT_ID=C...
   ```

5. **Run the app**:
   ```bash
   npm run dev
   # http://localhost:3000
   ```

### Live flow

On `/resale`, when `NEXT_PUBLIC_FACEVALUE_CONTRACT_ID` is set:
- A **Connect** button appears (Stellar Wallets Kit: Freighter, xBull, Albedo, Lobstr, Hana, Ledger).
- Connect → select **Fair $115** → *Generate proof & submit* → then **"Submit on-chain (wallet signs)"**.
- The wallet requests a signature. Accept it.
- The real `verify_resale` transaction is submitted to testnet RPC.
- The verdict shows the actual tx hash on Soroban testnet.
- The old ticket's nullifier is burned on-chain, preventing double-sell.

**Anti-double-sell, live:** submit the same demo ticket a second time → the contract reverts with `NullifierAlreadyUsed`. This is intentional; the on-chain nullifier set enforces it.

### Graceful degradation (no contract)

With `NEXT_PUBLIC_FACEVALUE_CONTRACT_ID` unset or empty:
- No **Connect** button is shown.
- The resale verdict uses the existing labelled **simulated** verdict with a derived hash.
- The app has full feature-parity: users can still generate proofs, see accept/deny, and move to the door-scan flow.
- This mode requires no network setup and lets you demo the app instantly.
